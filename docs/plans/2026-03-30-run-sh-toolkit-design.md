# run.sh — Svalbard Drive Toolkit

**Status:** Design
**Date:** 2026-03-30

## Overview

A modular, zero-dependency shell script that acts as the single entry point for
using a Svalbard drive. Plug in the USB stick, run `./run.sh`, and get a menu
of everything you can do with the data on it.

Replaces the current `serve.sh` which only starts servers. The new `run.sh` is a
thin launcher that discovers modular action scripts and presents them as a menu.

## Architecture

### Drive Layout

```
drive-root/
  run.sh                              thin launcher (~40 lines)
  .svalbard/
    entries.tab                       baked menu definition (generated at build time)
    actions/
      browse.sh                       list & open ZIM files in Kiwix
      maps.sh                         start tile server + open map viewer
      chat.sh                         pick model, start llama-server
      apps.sh                         open a web app (CyberChef, SQLiteViz, etc.)
      serve-all.sh                    start all services at once
      share.sh                        serve on LAN for others
      inspect.sh                      list all drive contents with sizes
      verify.sh                       check SHA-256 checksums against manifest
      provision.sh                    install host tools via brew/apt
    lib/
      platform.sh                     detect OS/arch
      binaries.sh                     find_binary (bundled or PATH)
      ports.sh                        find free port
      ui.sh                           colors, menu rendering
      process.sh                      PID tracking, cleanup trap
  bin/
    macos-arm64/
      kiwix-serve, go-pmtiles, llama-server, toybox, dufs, fzf, 7z, ...
    linux-x86_64/
      ...
    linux-arm64/
      ...
```

### Source of Truth

Action scripts live in `recipes/actions/` in the svalbard repo. At build time
(`svalbard sync`), the system:

1. Copies all action scripts to `.svalbard/actions/` on the drive
2. Copies lib scripts to `.svalbard/lib/`
3. Generates `entries.tab` with concrete labels based on actual drive content
4. Writes `run.sh` to the drive root

No runtime detection or conditional logic. The menu is fully baked at build time.

### entries.tab Format

Tab-separated, grouped by section headers. Generated by svalbard based on what
content and tools are present on the drive.

```
# Svalbard · finland-128 — generated 2026-03-30
# Format: label<TAB>script<TAB>args

[browse]
Browse encyclopedias — 24 ZIM files	.svalbard/actions/browse.sh
Browse videos — 1 collection	.svalbard/actions/browse.sh	video

[maps]
View maps — Finland basemap + 4 overlays	.svalbard/actions/maps.sh

[ai]
Chat with Qwen 3.5 35B-A3B	.svalbard/actions/chat.sh

[apps]
Open CyberChef	.svalbard/actions/apps.sh	cyberchef
Open SQLiteViz	.svalbard/actions/apps.sh	sqliteviz
Open DuckDB Shell	.svalbard/actions/apps.sh	duckdb-wasm

[data]
Query drug database — Fimea	.svalbard/actions/data.sh	fimea
Query dictionary — Joukahainen	.svalbard/actions/data.sh	joukahainen

[tools]
Encrypt/decrypt files — age	.svalbard/actions/tools.sh	age

[serve]
Serve everything	.svalbard/actions/serve-all.sh
Share on local network	.svalbard/actions/share.sh

[info]
List drive contents	.svalbard/actions/inspect.sh
Verify checksums	.svalbard/actions/verify.sh
```

Sections with no content are simply omitted. A default-128 drive without maps or
models would have no `[maps]` or `[ai]` section.

### run.sh Launcher

Approximately 40 lines. Responsibilities:

1. Resolve own directory (DRIVE_ROOT)
2. Source `.svalbard/lib/ui.sh`
3. Parse `.svalbard/entries.tab` into arrays
4. Render menu (using fzf if available in bin/, falling back to numbered list)
5. Execute selected action script with args
6. Loop back to menu on completion

No platform detection, no content scanning, no conditional logic.

The browse action (`browse.sh`) shows a submenu of ZIM files grouped by category.
Content and sizes are baked into the script at build time.

### Share on Network

Two modes, both binding to 0.0.0.0:

**Share services** (`serve-all.sh --network`): Starts Kiwix, tile server,
llama-server, web apps — all bound to 0.0.0.0 instead of localhost. Prints
LAN IP address for sharing.

**Share files** (`share.sh`): Starts dufs on the drive root. Provides a nice
directory listing UI with resumable downloads. Anyone on the network can browse
and download ZIM files, databases, etc. to their own device.

```
  Sharing drive on local network:

  http://192.168.1.42:8080

  Tell others to open this address in their browser.
  Press Ctrl+C to stop.
```

## Bundled Binary Toolkit

Small cross-platform binaries included on every drive in `bin/<platform>/`.

### Must-have (~11 MB per platform)

| Binary | Size | Purpose |
|--------|------|---------|
| toybox | ~1 MB | 200+ POSIX utilities: httpd, sha256sum, grep, sed, find, wget, tar, nc, stat, du, base64, xxd, hexedit |
| dufs | ~3 MB | HTTP file server with directory UI, resumable downloads, upload support. Used by share action. |
| fzf | ~3 MB | Fuzzy finder. Makes run.sh menu selection fast and searchable. |
| age | ~3 MB | Already included. File encryption/decryption. |
| 7z | ~1 MB | Universal archive tool: zip, 7z, tar, gz, xz, rar, iso. Fills toybox's missing zip/unzip. |

### Nice-to-have (~2 MB per platform)

| Binary | Size | Purpose |
|--------|------|---------|
| zstd | ~2 MB | Fast compression/decompression. Modern gzip alternative, better ratios. |

### Opt-in for larger presets (512 GB+)

| Binary | Size | Purpose |
|--------|------|---------|
| ffmpeg | ~30-80 MB | Universal media handling. Convert/play audio, video, images. |

### Per-platform total

Base toolkit: ~13 MB per platform.
Three platforms (macos-arm64, linux-x86_64, linux-arm64): ~39 MB total.
Negligible vs. drive content.

### Platform Builds

Platforms to support:

- `macos-arm64` — Apple Silicon Macs (primary target)
- `linux-x86_64` — most Linux desktops/laptops
- `linux-arm64` — Raspberry Pi, ARM laptops

All binaries should be statically linked where possible. toybox and 7z compile
easily as static. dufs and fzf distribute official static binaries. age already
ships static releases.

Note: toybox has limited macOS support (community forks exist). Fallback is to
rely on macOS built-in equivalents which are consistent enough.

Windows: out of scope. Users should use WSL.

### Recipes and Preset Inclusion

Each toolkit binary is a proper recipe in `recipes/tools/`, following the same
pattern as `kiwix-serve.yaml` and `age.yaml`:

```
recipes/tools/toybox.yaml
recipes/tools/dufs.yaml
recipes/tools/fzf.yaml
recipes/tools/7z.yaml
recipes/tools/zstd.yaml
```

Presets reference them like any other source. The base toolkit (toybox, dufs, fzf,
7z) is included in every preset from default-32 upward:

```yaml
# Every preset includes the base toolkit
- kiwix-serve
- age
- toybox
- dufs
- fzf
- 7z
```

Larger presets add optional binaries:

```yaml
# 128 GB+ presets add zstd
- zstd

# 512 GB+ presets add ffmpeg
- ffmpeg
```

This means the toolkit composition is controlled entirely through preset YAML
files — no special-casing in the build pipeline.

## Adding New Actions

To add a new action to the system:

1. Create a shell script in `recipes/actions/` in the svalbard repo
2. The script receives `DRIVE_ROOT` as an environment variable
3. The script can `source "$DRIVE_ROOT/.svalbard/lib/platform.sh"` etc.
4. Update the generator to include it in `entries.tab` when relevant content exists

For custom per-drive actions (e.g. a recipe ships a custom tool):

1. Recipe defines a `custom_action:` field in its YAML
2. Build step writes the script to `.svalbard/actions/custom-<id>.sh`
3. Build step appends an entry to `entries.tab`

## Migration from serve.sh

- `serve_generator.py` is replaced by a new `toolkit_generator.py` (or similar)
- The new generator copies action scripts, lib scripts, builds `entries.tab`,
  writes `run.sh`
- `serve.sh` is no longer generated. `run.sh` subsumes its functionality.
- The "Serve everything" action in run.sh does exactly what serve.sh did.

## Future: Search Index

A later addition: at build time, svalbard extracts titles and summaries from ZIM
metadata and SQLite content into a `search.sqlite` FTS5 database. A new
`search.sh` action queries it via the bundled toybox/sqlite3 and shows which
source contains relevant content, then offers to open that viewer.

This is a separate piece of work. The action script infrastructure supports it —
just drop a `search.sh` in `recipes/actions/` when ready.

## Implementation Sequence

1. Create `recipes/actions/` directory with initial action scripts
2. Create `recipes/actions/lib/` with shared helpers extracted from current serve.sh
3. Write the `entries.tab` generator in python (part of svalbard sync)
4. Write the `run.sh` template (thin launcher)
5. Add binary download recipes for toybox, dufs, fzf, 7z, zstd
6. Update `svalbard sync` to assemble `.svalbard/` on the drive
7. Remove `serve_generator.py`, update `readme_generator.py` to reference run.sh
8. Test on macOS and Linux
